# Application TCO — How IBM Apptio Costing Standard Addresses This TBM Capability

## The capability

Application Total Cost of Ownership answers a question every CIO eventually has to answer in
front of a CFO: what does each application actually cost to run, end to end — not just its
license, but the labor, infrastructure, vendor, and facilities cost that support it? Without a
defensible answer, application rationalization decisions (retire it, modernize it, consolidate
it) get made on gut feel instead of evidence, and legacy technical debt quietly compounds.

## How IBM Apptio Costing Standard covers it

Costing Standard's Applications capability catalogs every application across the enterprise and
combines labor, infrastructure, software, vendor, and facilities costs into a single defensible
TCO view per application. The documentation is direct about the problem this solves: "Many
enterprises struggle with limited application portfolio discipline, rising technical debt,
redundant applications, and poor cost transparency, often compounded by reliance on spreadsheets
and fragmented data."

This is delivered through a collection of five purpose-built reports, each aimed at a different
persona and decision:

- **Application Review** — executive-level view of top applications and cost trends, aimed at
  application owners and CIO/IT leadership.
- **Application Portfolio** — expands to the entire portfolio: who's responsible for what,
  spend by business unit, cost drivers by IT tower and vendor.
- **Application List** — per-application detail for application owners, including SaaS provider
  spend and Run vs. Development cost split.
- **Infrastructure Analysis by Application** — breaks down which infrastructure (compute,
  storage, cloud, service desk) supports each application, including on-prem vs. public cloud
  proportion.
- **New & Retired Applications** — tracks portfolio churn over the trailing 12 months, useful
  for measuring actual rationalization progress against a stated goal.

A sixth report, **Application Model**, exists specifically for traceability — it shows how cost
data moves through the Apptio model (Allocation Models, Tower/Sub-Tower structure, Cost Pools) so
an analyst can validate and troubleshoot a specific application's number rather than just trust it.

## Why this matters in a client conversation

The persona list is worth calling out explicitly: this isn't only an IT Finance tool. Enterprise
Architecture uses the same view to guide standardization and consolidation decisions, and
Business Relationship Managers use it to have accountable consumption conversations with the
business side. That breadth is the pitch — one TCO number, reused consistently across the
audiences that all currently argue about a different, less defensible number.

The custom use cases the documentation calls out (Application Lifecycle Management, Technology
Stack Consolidation) are where this ties directly into rationalization initiatives — a common RFP
ask that's really just "give us application TCO plus a lifecycle lens on top."
