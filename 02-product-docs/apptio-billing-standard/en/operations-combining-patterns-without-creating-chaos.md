# Combining patterns without creating chaos

Most real implementations end up as a mix, for example:

- Service catalog centric plus chargeback, with a bit of project tagging.
- Cloud centric for cloud domains and simple showback for everything else.
- Service catalog centric plus legal entity view for only a subset of services.

Practical guardrails:

- Decide which pattern is “primary.” That pattern should drive how you name services, define units, and structure reports.
- Use additional patterns to extend visibility, not to create parallel competing billing schemes.
- Document which patterns are active, for which services, and for which audiences.

If the core model is clean and documented, these patterns become powerful lenses on the same data instead of separate, conflicting stories.
